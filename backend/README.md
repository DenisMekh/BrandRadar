# BrandRadar API

SaaS-платформа мониторинга репутации брендов. Собирает упоминания из Web, RSS и Telegram, анализирует тональность через ML-сервис, детектирует аномалии и отправляет алерты в Telegram.

## Ключевые пути данных

### 1. Пайплайн сбора упоминаний

```
Источник (web/rss/tg) → Воркер аналитики (каждую 1 мин)
  → Crawler.Crawl()              // параллельный HTTP-запрос + goquery-парсинг
  → Deduplicator.Filter()        // дедупликация через Redis по хэшу текста
  → ML.CheckRelevance()          // совпадение с ключевыми словами бренда
  → ML.AnalyzeSentiment()        // positive / neutral / negative + confidence
  → Repo.InsertCrawlerItems()    // сохранение в БД
  → EventLog: mention_created
```

### 2. Детекция аномалий → Алерт

```
Воркер аномалий (каждые 2 мин)
  → Получить все активные AlertConfig
  → Для каждого бренда:
      → collectDataPoints()      // взвешенный перцентиль тональности за окно
      → Analyze()                // доверительный интервал (μ ± 1.96σ)
      → если аномалия:
          → AlertUseCase.CheckAndFire()   // проверка cooldown + запись алерта
          → TelegramNotifier.Send()       // отправка в чат
```

### 3. API-запрос → Дашборд

```
GET /api/v1/brands/{id}/dashboard
  → DashboardUseCase.GetByBrand()
  → Repo: подсчёт тональностей (pos/neg/neutral)
  → Repo: разбивка по источникам
  → Repo: дневная временная шкала
  → Repo: количество недавних алертов
  → Response DTO
```

## Компромиссные решения (Tradeoffs)

### 1. Нет полноценного краулера

Текущий веб-краулер — это goquery-парсер с жёстко заданными CSS-селекторами (`article > h2 a`, `.meta`, `p`).

**Что не умеет:**
- JS-рендеринг (SPA-сайты)
- Пагинация, авторизация
- Rate limiting с retry, соблюдение robots.txt

Telegram-краулер парсит HTML-превью каналов, что хрупко и ломается при смене вёрстки.

**Причина:** скорость разработки. Полноценный краулер (Colly, chromedp, Scrapy) — это отдельный микросервис со своей очередью задач.

**Влияние:** ограниченное покрытие источников, хрупкость при изменении вёрстки сайтов.

### 2. Упрощённый доверительный интервал для аномалий

Используется наивный Z-score с фиксированным порогом λ = 1.96 (95% доверительный интервал):

```
μ = mean(window)
σ = stddev(window)
аномалия = value < μ − 1.96σ  ИЛИ  value > μ + 1.96σ
```

**Ограничения:**
- Предполагает нормальное распределение — реальные данные часто имеют тяжёлые хвосты
- Нет учёта сезонности (день/ночь, будни/выходные)
- Фиксированный порог без адаптации к волатильности бренда
- Population stddev (деление на N, а не N−1) — занижает дисперсию на малых выборках
- При σ = 0 любое отклонение считается аномалией (Z = ∞)

**Причина:** простота реализации и достаточность для MVP. Альтернативы (ARIMA, Prophet, Isolation Forest) требуют отдельного Python-сервиса.

**Влияние:** возможны ложные срабатывания при выбросах и пропуски при медленных трендах.
